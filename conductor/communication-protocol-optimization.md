# Communication Protocol Optimization: Structured Agent Messaging

**Date:** 2026-04-11  
**Author:** ai-colab Conductor Agent  
**Classification:** Strategic Architecture Document  
**Status:** Proposal for Review  

---

## Executive Summary

Currently, all inter-agent communication in ai-colab uses plain English via hcom messaging. While this ensures human readability, it introduces significant inefficiencies:

1. **Context bloat** — LLM "thinking" output, conversational filler, and redundant explanations consume blackboard storage and event log capacity
2. **Processing overhead** — Every agent must parse natural language to extract actionable data
3. **Ambiguity** — Natural language is imprecise for structured data (percentages, timestamps, error codes)
4. **Event log pollution** — The conductor's event cursor processes every message, including non-actionable "thinking" output

**Recommendation:** Implement a **dual-layer communication protocol** — structured JSON for agent-to-agent data exchange, with human-readable summaries generated on-demand for the conductor dashboard and Web UI.

```

     │
     ↓ Conductor processes this as an event, stores in event cursor,
       agent reports health status separately
```

### Inefficiencies Identified

| Issue | Impact | Example |
|-------|--------|---------|
| **Thinking output in messages** | 60-80% of message tokens are non-actionable | "Let me think about this... I should consider..." |

| **Redundant status updates** | Same information repeated across messages | Multiple agents reporting "I'm working on X" |
| **No structured error codes** | Errors buried in text, hard to detect programmatically | "Hmm, that didn't work, let me try something else" |
| **Blackboard storage bloat** | JSON payloads in text fields, no schema validation | `agent_task_gemini: "Working on implementing the new feature..."` |

## Proposed Architecture: Dual-Layer Protocol

### Layer 1: Structured Agent-to-Agent (Machine-Optimized)

All automated communication uses a compact, structured format:

```json
{
  "v": 1,
  "t": "status",
  "a": "gemini",
  "ts": 1712876400,
  "track": "implement-api",
  "phase": "coding",
  "pct": 45,
  "step": "writing endpoint handlers",
  "eta": 1800,
  "blockers": []
}
```

**Field definitions:**
- `v` — Protocol version (int)
- `t` — Message type: `status`, `request`, `response`, `error`, `complete`
- `a` — Agent name (string)
- `ts` — Timestamp (Unix epoch)
- `track` — Current track slug (string)
- `phase` — Current phase: `analyzing`, `coding`, `testing`, `reviewing`
- `pct` — Progress percentage (0-100)
- `step` — Current step description (short string, max 100 chars)
- `eta` — Estimated seconds to completion (int)
- `blockers` — Array of blocker descriptions (strings)
- `err` — Error code (string, optional)
- `detail` — Human-readable detail (string, optional — only for dashboard)

**Size:** ~150 bytes vs. ~1,500 bytes for equivalent English message = **90% reduction**

### Layer 2: Human-Readable Summaries (On-Demand)

The conductor generates human-readable summaries from structured data:

```
Structured message:
{"t":"status","a":"gemini","track":"implement-api","pct":45,"step":"writing endpoint handlers","eta":1800}

Human-readable summary (generated by conductor dashboard):
  Gemini: 45% complete on implement-api — writing endpoint handlers (~30 min remaining)
```

**Key principle:** Humans see summaries when they need them. LLMs exchange structured data always.

### Protocol Message Types

| Type | Purpose | Required Fields |
|------|---------|-----------------|
| `status` | Progress update | `track`, `pct`, `step` |
| `request` | Task handoff or help request | `track`, `phase`, `detail` |
| `response` | Reply to request | `track`, `detail` |
| `error` | Failure report | `track`, `err`, `detail` |
| `complete` | Task completion | `track`, `pct: 100` |
| `heartbeat` | Health check | `a`, `ts` |

## Implementation Plan

### Phase 1: Message Format Definition

**Files:** `config/message-protocol.json`

Define the schema and message types. Agents can validate outgoing messages against this schema.

### Phase 2: Agent Wrapper Integration

**Files:** `scripts/agent-wrapper.sh`, `scripts/protocol-encoder.sh`, `scripts/protocol-decoder.py`

- `protocol-encoder.sh` — Convert structured data to hcom message format
- `protocol-decoder.py` — Parse incoming messages, extract structured data
- Agent wrapper captures LLM output, filters "thinking" content, extracts structured status

### Phase 3: Conductor Protocol Handler

**Files:** `scripts/conductor-workflow.sh`, `scripts/protocol-handler.py`

- Conductor processes structured messages natively (no NLP parsing needed)
- Generates human-readable summaries for dashboard display
- Detects errors automatically via `err` field
- Tracks progress numerically via `pct` field

### Phase 4: Blackboard Schema Update

**Files:** `config/blackboard-schema.json`

Add namespaces for structured protocol data:
- `agent_protocol_<name>` — Latest protocol message per agent
- `protocol_errors` — Active error queue
- `protocol_stats` — Message throughput, error rates

### Phase 5: Backward Compatibility

- Existing English messages still processed (with lower priority)
- Gradual migration: agents can send both structured and English simultaneously
- Dashboard shows structured data when available, falls back to English parsing

## Human Override Mechanism

### Request Human-Readable Mode

Any agent or human operator can request full English output:

```
hcom send @all --name conductor --intent request --thread "protocol" -- \
  '{"t":"request","mode":"verbose","agent":"gemini"}'
```

The targeted agent switches to verbose English mode for its next N messages (configurable, default: 5 messages).

### Conductor Dashboard Toggle

The tmux dashboard includes a toggle:
- `Ctrl+b v` — Toggle verbose mode (show full agent output)
- `Ctrl+b c` — Compact mode (structured summaries only)

## Sample Implementation: Agent Wrapper Integration

```bash
# In agent-wrapper.sh, after LLM CLI execution:
# Capture the agent's output, filter thinking, extract structured status

capture_agent_status() {
    # If the agent outputs structured JSON, use it directly
    if echo "$agent_output" | jq -e '.t' >/dev/null 2>&1; then
        echo "$agent_output"
        return 0
    fi

    # Otherwise, generate structured status from known state
    cat << EOF
{
  "v": 1,
  "t": "status",
  "a": "$HCOM_NAME",
  "ts": $(date +%s),
  "track": "$CURRENT_TRACK",
  "phase": "$CURRENT_PHASE",
  "pct": $CURRENT_PROGRESS,
  "step": "$CURRENT_STEP",
  "eta": $CURRENT_ETA,
  "blockers": [],
  "detail": "Status auto-generated by agent wrapper"
}
EOF
}
```
